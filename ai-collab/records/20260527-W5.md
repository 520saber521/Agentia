## 2. 关键决策

### 决策 1：对话式 Agent 创建走自然语言解析

**背景**：REBUILD_PLAN W5-T1 要求 `POST /api/agents` + AgentManage 表单页。但 SPEC 描述的是"对话式创建"。

**选择**：两路都做了。`AgentCreateDialog.tsx` 提供表单式精确配置；`_parse_agent_create_request()` 让用户在群聊里直接输入 `/agent create name: 前端审查员 | adapter: claude_code | prompt: 你是一个资深前端 review 专家` 创建 Agent。

**收益**：表单式满足"精确配置"，对话式满足"快速创建"——即使用户在聊天中途突然想到需要一个新角色，不用跳出会话去管理页。

### 决策 2：Pin 上下文按"最新优先 + 截断提示"注入

**背景**：一个长会话可能 pin 了 20 条消息，全部注入 prompt 会超过 8K token 预算。

**选择**：BFF 在调用 Adapter 前收集 pinned 消息，按 `created_at DESC` 排序取前 5 条，拼成 `Pinned Context` 块注入到 system prompt 区域。超过 5 条时追加一句 "…以及 N-5 条较早的 pinned 消息（未注入）"。

**收益**：Agent 不会因为"pin 了太多"而丢失关键上下文，用户也知道有多少被截断了。

### 决策 3：P2 项的显式降级标注

**背景**：部署发布和多端支持标注为 P2，但仍需在最终交付物中体现。

**选择**：部署只做本地静态预览 URL（`/preview/{artifact_id}` 挂载），多端仅保证 1024px 以上可用。两个 P2 项在前端 UI 上都有入口按钮但标注"P2 降级"。

**收益**：不会在大答辩时被问到"部署在哪"而仓促解释——入口已经展示了意图和边界。

## 3. 与 AI 协作的真实片段

### 3.1 `/agent create` 自然语言解析的正则打磨

对话式 Agent 创建一开始的 prompt：

> 我在聊天框里打 `/agent create name:前端审查员 | adapter:claude_code | prompt:...` 就能创建一个 Agent。

AI 第一版用 `re.split(r'[,;|]', body)` 切分字段，但遇到 system_prompt 内容本身包含 `|` 时就会切错。我指出后 AI 改成正则 `(\w+)[:：]\s*(.+?)(?=\s+\w+[:：]|$)` 做贪婪匹配。但这个正则在 capability 字段的值里有 `/` 时又挂了。

最终方案是先用 `|`/`;`/换行切分 `key: value` 对，再逐对解析。虽然不是完美的通用解析器，但覆盖了 90% 的真实输入。

> 反思：自然语言解析的边界 case 永远比预想的多。SPEC 里应该写一条反例："system_prompt 内容包含 `|` 或换行时，需要通过配置表单而非对话式创建"。

### 3.2 `ORCHESTRATOR_SYSTEM_PROMPT` 常量失踪案

W5-D3 我重构 `services/agent.py` 时删掉了 SQLite 依赖相关的旧常量，但不小心把 `ORCHESTRATOR_SYSTEM_PROMPT` 也删了。结果 `db/seed.py` 导入失败，连锁导致 `test_db.py` 和 `test_w2_fan_out.py` 两个测试文件直接 collection 阶段崩溃。`adapter_config_for_runtime` 和 `record_agent_execution_start/finish` 也有同样的问题——这些函数在 `send_message.py` 中被使用但从未在 `services/agent.py` 中定义。

根因：W3/W4 开发时这些函数被临时写在 `handlers/__init__.py` 或 `main.py` 的 inline 位置，W5 重构时被移动到了 `services/agent.py`，但移动时漏了三个符号。

> 反思：**任何跨文件 import 的重构必须先 grep 所有引用点，再移动定义，再跑全量测试。** 这条应该写进 `ai-collab/rules/backend.mdc`。
