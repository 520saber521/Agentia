## 2. 本次解决的具体问题

### 问题 1：后端缺少 `update_conversation` 和 `delete_conversation` 函数

**问题描述**：REST 路由引用了 `services.conversation.update_conversation` 与 `delete_conversation`，但 service 层未实现这两个函数，导致 BFF 启动时抛出 `ImportError`。

**解决方案**：
- 在 [conversation.py](server/services/conversation.py) 中新增 `update_conversation()` 函数，支持更新 `title` / `pinned` / `archived` 字段。
- 新增 `delete_conversation()` 函数，利用 SQLAlchemy ORM 级联删除（`ondelete="CASCADE"`）清理关联的 `ConversationMember` 与 `Message` 记录。
- 在 [rest.py](server/api/rest.py) 中注册 `DELETE /api/conversations/{conversation_id}` → 204 路由，404 时返回 HTTPException。

**实施效果**：BFF 正常启动，删除会话接口返回 204 No Content，关联数据完整清理。

---

### 问题 2：Vite 代理端口不匹配导致前端离线

**问题描述**：`vite.config.ts` 代理目标配置为 `8999` 端口，但后端实际运行在 `8788` 端口；前端端口配置为 `5174` 但实际使用 `5173`。所有 API 请求返回 `ECONNREFUSED`，前端显示离线状态。

**解决方案**：修正 [vite.config.ts](web/vite.config.ts) 中 `proxy.target` 为 `http://127.0.0.1:8788`，`server.port` 为 `5173`。

**实施效果**：前端 API 请求正确代理到后端，WebSocket 连接正常建立。

### 问题 3：HTML 代码块以纯文本显示，无法渲染预览

**问题描述**：Agent 回复中的 `` ```html...``` `` 代码块被 `TextBubble` 统一解析为语法高亮的 `CodeBlock` 组件，用户无法看到实时渲染效果。

**解决方案**：
- 在 [TextBubble.tsx](web/src/components/ContentRenderer/TextBubble.tsx) 中扩展解析器，新增 `HtmlPart` 类型识别语言标记为 `html` / `htm` 的代码块。
- 内联实现 `InlineHtmlPreview` 组件：使用 `<iframe srcDoc={code} sandbox>` 安全渲染 HTML 内容；顶部工具栏提供"预览/查看源码"切换按钮；橙色边框标识预览区域。

**实施效果**：`` ```html `` 代码块自动渲染为实时 iframe 预览，高度 480px，支持脚本执行与样式渲染。

### 问题 4：前端组件 Props 缺失导致运行时崩溃

**问题描述**：
- `App.tsx` 中 `<ConversationListPanel />` 未传入任何 props，导致 `conversations` 为 `undefined`，`.map()` 触发 `TypeError: Cannot read properties of undefined (reading 'map')`。
- `App.tsx` 未从 store 解构 `conversations` 变量，触发 `ReferenceError: conversations is not defined`。

**解决方案**：
- 在 [App.tsx](web/src/App.tsx) 中添加 `const conversations = useChatStore((s) => s.conversations)` 解构。
- 补全 `<ConversationListPanel conversations={conversations} currentId={currentConvId} onNewAgent={() => setAgentDialogOpen(true)} />` props 传递。
- 在 [ConversationListPanel.tsx](web/src/components/ConversationListPanel.tsx) 参数声明中设置 `conversations = []` 默认值作为防御性编程。

**实施效果**：左侧面板正常渲染，不再出现白屏或控制台报错。

### 问题 6：群聊多 Agent 协作失败

**问题描述**：
- 群聊创建时未自动加入 Orchestrator，无法进行任务拆解编排。
- 无 API Key 的 Codex 类 Agent 收到消息后返回错误，污染聊天流。
- 群聊无 @mentions 且文本不够"复杂"时返回 `no_target` 错误，拒绝响应。

**解决方案**：
- [conversation.py](server/services/conversation.py)：创建 `type="group"` 会话时自动将 `ORCHESTRATOR_AGENT_ID` 插入成员列表首位。
- [send_message.py](server/handlers/send_message.py)：新增 `_filter_available_agents()` 函数，按规则过滤：
  - `mock` / `claude_code` / `opencode` → 无需 Key，始终可用
  - 有 `api_key` 的 Agent → 可用
  - 无 `api_key` 的 codex Agent → 静默跳过
  - Orchestrator → 始终保留
- 群聊无 @mentions 时改为广播给所有可用 Agent（而非返回错误）。

**实施效果**：群聊发消息后 Orchestrator 自动参与任务拆解，仅可用的 Agent 响应，无 Key Agent 不产生错误噪音。

### 问题 7：seed 函数 UNIQUE constraint 冲突导致后端崩溃

**问题描述**：`seed_demo()` 中对 `conv_demo` 执行无条件 INSERT INTO conversation_member，重复启动时触发 `UNIQUE constraint failed: conversation_member.conversation_id, member_id`，导致 BFF 启动失败退出。

**解决方案**：[seed.py](server/db/seed.py) 改为先查询已存在的 member 记录，仅在缺失时执行 INSERT。

**实施效果**：后端热重载后稳定重启，不再因 seed 冲突崩溃。

## 3. 关键决策

### 决策 1 · DeepSeek 采用 OpenAI 兼容协议适配器

DeepSeek API 提供标准 OpenAI 兼容的 `/chat/completions` 端点（SSE 流式），因此选择独立实现 `DeepSeekAdapter` 而非复用通用 Codex 适配器。这样做的好处是：
- 可以针对 DeepSeek 特有参数（如 `deepseek-reasoner` 推理模型）做定制化处理；
- 未来若 DeepSeek 推出工具调用（function calling）等高级能力，可在本适配器中独立扩展，不影响其他 adapter。

### 决策 2 · HTML 预览内联于 TextBubble 而非独立路由

选择将 `InlineHtmlPreview` 作为 TextBubble 内部的一种 Part 渲染方式，而非跳转到独立页面或弹窗。原因：
- 保持聊天流的上下文连贯性，用户无需离开当前对话即可预览；
- 利用 iframe sandbox 隔离安全性，避免 XSS 风险；
- 预览与源码一键切换，兼顾"所见即所得"与"审查代码"两种需求。

### 决策 3 · 双 Tab 架构分离 Agents 与 Chats

将左侧面板分为 Agents Tab 和 Chats Tab，而非混合展示。理由：
- 用户心智模型中"选 Agent"和"看历史"是两个不同操作意图；
- Agents Tab 支持多选建群的操作空间更大（复选框 + 操作栏）；
- Chats Tab 的删除等管理功能不会误触 Agent 相关操作。

## 5. 已知边界

- 当前 HTML 预览不支持跨域资源加载（如外部 CDN 图片/字体），iframe sandbox 限制了部分权限；后续可考虑增加"信任模式"放开限制。
- 群聊广播模式下各 Agent 独立回复，尚未实现回复聚合（merge/reduce）；该能力可由 Orchestrator 的 `handle_orchestrator_mention` 进一步扩展。
- DeepSeek Adapter 当前仅支持 text 输出，尚未接入 function calling / tool_use 能力；待 DeepSeek 官方 API 稳定后可扩展。
- 删除操作在前端 try/catch 包裹下即使后端失败也会从 UI 移除，可能导致前后端短暂不一致；后续应考虑乐观回滚机制。
