## 2. 5 个关键决策

### 决策 1 · 错误码作为业务契约的一等公民

**背景**：W1 `POST /api/conversations` 用 `ValueError(str)` 抛错，REST 层一律映射为 `400 detail=<str>`，前端基本只能展示原始字符串。

**问题**：F-W2-5 SPEC 显式约定了 4 个错误码（`title required` / `invalid_type` / `group_requires_agents` / `unknown_agent`），如果前端按子串匹配文案就脆弱。

**选择**：

- service 层抛 `ValueError(code)`，**code 用稳定枚举（小写下划线）**，与 SPEC F-W2-5 / `ai-collab/rules/backend.mdc` R-B-6 对齐。
- REST 层 `_VALUE_ERROR_HTTP_STATUS` 表把 code 映射到 HTTP 状态（默认 400，已知 4 码映射到 422）。
- 前端 `ApiError.code` 类型化为字面量联合；`describeApiError(err)` 给每个 code 一句中文。

**收益**：后续新增任何 4xx 错误，只需要：service 抛 code + 表里加一行 + 前端 `ApiErrorCode` 联合加一项。**0 处需要改 UI 字符串匹配逻辑**。

**沉淀**：在 `ai-collab/rules/backend.mdc` R-B-6 现有条款之外，明天 W2-D2 启动 F-W2-1 时把"业务错误 code 必须是稳定小写枚举"显式升格成 R-B-6 的一个子项（候选条款）。

---

### 决策 2 · 错误码白名单 + `unknown` fallback

**背景**：前端 `ApiError.code` 想要 TypeScript 字面量类型保护，但后端总有"还没纳入白名单"的 detail。

**取舍**：

- ❌ 选项 A：让 `code: string`，TypeScript 失去保护。
- ❌ 选项 B：让前端强校验，未知 detail 抛二级错误（太脆，会让稍微落后于后端发布的前端整页崩）。
- ✅ 选项 C：白名单 + `"unknown"` fallback；UI 在 unknown 时降级展示原 `detail`。

**收益**：

1. TypeScript 在 `switch (err.code)` 里强制穷举，新增错误码会立刻 lint 失败 → 强制 PR 作者同步前端文案。
2. 前后端版本错配时前端仍能展示**人类可读**的原文，不会让用户看到一个空白错误。

---

### 决策 3 · 抽 `_upsert_agent`，不直接复制粘贴 seed 代码

**背景**：W1 `seed.py` 用 `if row is None: s.add(...)` 写过一次 `Agent` upsert。W2 起多了 `agent_mock_2`，未来 W2-T2 还要加 `agent_claude`，W3 加 `agent_codex` / `agent_orchestrator`，每个都重复 6 行代码。

**选择**：抽 `_upsert_agent(s, agent_id, fields)`，让 seed 主流程读起来像声明而非命令。

**反例**（拒绝的选项）：

- 拒绝写 `INSERT OR IGNORE` 之类 SQL hack —— SQLAlchemy `merge` / `add` 行为对幂等友好，但只在某些方言下，会让"看代码就知道结果"的可读性下降。
- 拒绝引入 `alembic.upgrade()` 风格的 migrations —— W2 还用不上 schema 变更，引入只增加心智负担。

**沉淀**：未来加 `agent_claude` / `agent_orchestrator` 时只调一个 `_upsert_agent`，不会有"漏改 if 分支"的风险。

---

### 决策 4 · 默认行为变更：`createConversation` 不再隐式塞 `agent_mock`

**背景**：W1 `client.ts` 默认 `agent_ids: ["agent_mock"]`，是为了让 W1 调试控制台无脑发送也能成。

**问题**：W2 F-W2-5 校验"group 必须 ≥1 agent"，如果 default 还是 `["agent_mock"]`，会让"没传 agent_ids"和"传了 [agent_mock]"在后端无法区分，**反而掩盖前端 UI 的 bug**。

**选择**：default 改为 `[]`，让 UI 必须显式选择。这条改动通过 `test_create_conversation_group_requires_agents_endpoint` 用例校验：传 `agent_ids: []` 必须 422。

**反例验证**：如果将来 NewConversationDialog 因为 bug 没把 selected 注入 body，请求就会被后端 422 拒绝，前端会展示"群聊需要至少 1 个 Agent"——bug 立刻可见，不会静默成功。

## 3. 与 AI 协作的真实片段

### 3.1 关于错误码"是 service 抛 code、还是 REST 抛 HTTPException"

我（Operator）的 prompt 隐含意图：让错误处理"统一而不啰嗦"。

AI 给的两个候选：

- **A 方案** · service 直接 `raise HTTPException(422, "group_requires_agents")`，REST 层透传。
  - 优点：路径短、不需要错误码映射表。
  - 缺点：service 层耦合到 HTTP 语义，未来如果换 gRPC / CLI 入口就要重写。

- **B 方案** · service 抛 `ValueError(code)`，REST 层 `_VALUE_ERROR_HTTP_STATUS` 映射。
  - 优点：service 保持框架无关；错误码集中映射，易审。
  - 缺点：多一层映射表。

我选 B（B 已写进 `ai-collab/rules/backend.mdc` R-B-2 的精神：DB 访问必须经过 services 层 —— 反向推论 services 不应反向依赖 HTTP）。

> **反思**：service 是"业务事实"，REST 是"展现"。FastAPI 写多了容易模糊这层边界。Cursor 第一版直接给的 A，被我打回后才给出 B。下次写新功能可以在 prompt 里**直接**点名"service 层不能依赖 FastAPI"，少走一次回合。

---

### 3.2 关于"要不要为 `agents.length === 0` 加专门的提示"

AI 第一版的 NewConversationDialog 在 `agents.length === 0` 时让"创建"按钮 disabled 但**没有任何文案**，会让用户以为系统死机。

我让它加：`{!loadingAgents && agents.length === 0 && (<div>暂无可选 Agent</div>)}`。

AI 又问要不要加一个"刷新"按钮。我**拒绝**：

> 没必要。`useEffect` 在模态打开时就拉一次；如果 `agents.length === 0` 是真实状态（如 W5 时用户删光了自建 Agent），刷新没用。**SPEC F-W2-5 没有"重新加载"这一行为，不擅自补 UI**。

> **反思**：AI 协作时最容易"为了周到而多做"。和 SPEC 对齐才是减少误判最快的方法。
