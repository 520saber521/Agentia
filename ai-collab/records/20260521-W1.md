## 2. 5 个关键决策（Why I chose this）

### 决策 1：W1 用 MockAdapter，不接真模型

**背景**：原计划 Day4 接 Claude，被 Token 申请流程卡住。
**选择**：把 `MockAdapter` 做扎实（按 token 切流 + 真支持 cancel），全 W1 都用它跑链路。
**收益**：所有跨层 bug（队列卡死 / 取消语义 / WS proxy 漏帧）在 W1 就被发现 + 固化为单测，W2 接真 API 时只需新建一个 Adapter 文件。
**对应规约**：`ai-collab/rules/adapter.mdc` R-A-3 / R-A-4 就是这一周踩出来的。

### 决策 2：BFF 用 SQLAlchemy 而不是直接拿 sqlite3

**背景**：W1 数据量小到 1000 行内，sqlite3 模块就够了。
**选择**：上 SQLAlchemy 2.x async + `aiosqlite`。
**收益**：services 层的"事务一致性"变成 `async with Session()` 这种很短的语法；W2 接群聊时改成 PostgreSQL 是 1 行连接串变更。
**代价**：Day3 踩了一个 `m.created_at` 在 commit 前是 None 的坑（详见决策 5），花了 ~30 分钟。

### 决策 3：WebSocket 用"单写者池"，不是直接多协程 send

**背景**：Day2 一开始我让 `_stream_reply` 协程和 `_handle_ping` 协程都直接 `await ws.send_json(...)`。
**触发**：跑 smoke_day2 时偶发出现 `cannot call "send" once a close message has been sent` —— 多协程在 socket close 后竞争。
**选择**：引入 `Connection` 类，所有发送入 `asyncio.Queue`，由一个 `writer()` task 串行 send。
**收益**：这成为 `ai-collab/rules/backend.mdc` R-B-4。后续任何"BFF 主动推某事件"的需求都不会再撞上 race。

### 决策 4：reducer 抽纯函数

**背景**：Day4 store 里 `handleEvent` 一开始混着 `console.error` / `set` / `void get().refreshConversations()`。
**触发**：Day5 写单测时发现根本测不动 —— Zustand store 起来要先 `init()`，`init()` 又会去连 WS。
**选择**：把状态映射抽成 `reducer.ts`，副作用以 `effects: SideEffect[]` 返回，由 store 层实际执行。
**收益**：15 个用例 4ms 跑完，覆盖了"非当前会话事件不污染 messages"这种之前靠脑补的不变式。

### 决策 5：所有 `created_at / updated_at` 由调用方显式 `now_ms()`

**背景**：Day3 跑 `test_create_message` 报 `TypeError: '>' not supported between instances of 'NoneType' and 'int'`。
**根因**：`Message.created_at` 列虽然有 `server_default`，但**在 commit/flush 之前** ORM 实例上还是 None。
**选择**：service 层先生成 `ts = now_ms()`，显式赋值给 `Message(...).created_at`，并传给 `_touch_conversation(ts=ts)`。
**沉淀**：`ai-collab/rules/backend.mdc` R-B-3。

## 3. 与 AI 协作的真实片段

### 3.1 触发 R-B-3 那次

我（Operator）的 prompt 大意：

> 跑 `test_create_message` 报 `TypeError: '>' not supported between instances of 'NoneType' and 'int'`，stack trace 指向 `_touch_conversation` 里的比较语句。这是 SQLAlchemy 的 server_default 还没回填的问题吗？请给我两个候选修法、并指出哪个有更长尾收益。

AI 给的两个方案：

- **A 方案**：在比较前 `await s.flush()` 强制回填 → 单点修复，但每个 service 都要记得加 flush。
- **B 方案**：service 层显式 `now_ms()` 生成 ts → 业务层永远拿到 int，不依赖 ORM 的回填时机。

我选 B 并明确要求"把这个写进 ai-collab/rules/backend.mdc"。这就是 R-B-3 的来历。

> 反思：如果不是在 ai-collab/ 里固化，下次同样的坑还会踩。这就是这一周最大的收获 —— **教训不能只活在 commit message 里**。

### 3.2 reducer 抽离那次

Day5 我让 AI "把 useChatStore 的 handleEvent 抽出来好测"。AI 第一版还是把 `console.error` 留在 reducer 里。我打回去，明确要求"reducer 必须纯，副作用通过返回值告诉调用方"。

AI 第二版补出了 `ReduceResult { next, effects }` 这个签名 —— **这个签名后来变成了 `ai-collab/rules/frontend.mdc` R-F-1**：所有事件→状态的映射必须经过 reducer。

> 反思：和 AI 共事很多次发现"代码风格清单 + 1-2 个反面例子"比"宏大的理念"更管用。后续 rules 全部按这个套路写：每条规则前面是"约束"，后面跟一个真实踩坑的引用。
