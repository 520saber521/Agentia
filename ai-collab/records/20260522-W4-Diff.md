## 2. 关键决策

### 决策 1 · 用 `before` 精确匹配判断过期

F-W4-5 要求对已过期 base 版本返回 `409 artifact_conflict`。本次同时做两层检查：

- 如果 base artifact 已有子版本，直接判定为过期。
- 如果客户端提交的 `before` 与当前 base 内容不一致，也返回 `artifact_conflict`。

这样可以防止用户在旧 Diff 卡片上重复应用，也能防止 UI 缓存与服务端文件内容不一致时悄悄覆盖。

### 决策 2 · 应用 Diff 后显式插入新产物消息

SPEC 要求“应用成功后在聊天流插入新的 `preview` 或 `code` 卡片”。因此后端不只返回 artifact，而是复用 message service 插入一条普通聊天消息，再广播 `message_created`。这样刷新后历史也能看到新版本，不依赖前端本地临时状态。

### 决策 3 · 补 `/api/artifacts/*` 而不是改前端回退

现有前端已经统一按 `/api/*` 调 REST，artifact 模块却暴露在 `/artifacts/*`。本次选择在后端给 artifact/upload/list/history/content/apply-diff 全部补 `/api` 别名，并保留旧路径，避免破坏已有手动调试脚本。
