## 2. 关键决策

### 决策 1：Artifact 独立表，消息 content 只存引用

**背景**：W3 的设计里代码块可以直接塞进 `message.content` JSON。当 Agent 输出 200 行 HTML 时，一条消息的 content 字段达到 15 KB，SQLite 查询明显变慢。

**选择**：遵循 REBUILD_PLAN P-5 —— 所有"可被预览/编辑/下载"的对象落 `artifact` 表，消息 `content` 仅存 `artifact_id` + 预览元数据。`_try_create_artifact()` 在 agent 回复完成后自动扫描代码块并创建 artifact。

**收益**：版本链天然成立（`parent_id` 回溯），artifact 单独查询不走 messages 大表。

### 决策 2：后端路径别名 `/api/artifacts/*` 而非改前端

**背景**：前端统一按 `/api/*` 调 REST，artifact 模块最初暴露在 `/artifacts/*` 下。

**选择**：在 `server/api/artifacts.py` 同时 mount 到 `/artifacts` 和 `/api/artifacts`，保留旧路径。

**收益**：零前端改动；已有调试脚本不破。

### 决策 3：CodeBlock 正则提取自动化

**背景**：Agent 返回的代码块在 ` ```language ... ``` ` 围栏内。如果每次都让前端解析 markdown，消息流内会有闪烁。

**选择**：在 `send_message.py` 的 `run_agent_reply()` 完成后用 `CODE_BLOCK_RE` 扫描 final_text，自动创建 artifact。artifact 创建后消息 content 从 `text` 变为 `code / preview`。

**代价**：正则匹配覆盖了 12 种语言，但不支持嵌套围栏（如 markdown 内的 markdown 代码块）。

### 决策 4：HTML 预览的截断恢复

**背景**：Claude / DeepSeek 等真实模型在 `max_tokens` 限制下经常输出不完整的 HTML（例如 `</style>` 或 `</html>` 缺失）。

**选择**：在 `orchestrator.py` 和 `send_message.py` 的 frontend subtask 路径上加入 `_close_partial_html()` —— 检测 `<!doctype html>` 位置，补上缺失的闭合标签，注入 fallback body。同时保留 `_visible_generation_error()` 在聊天气泡尾部标注截断原因。

**收益**：用户在 90% 的截断场景下仍能看到"可见但不完整"的预览，而不是空白 iframe。这是 W4 最被用户感知到的质量提升。

## 3. 与 AI 协作的真实片段

### 3.1 `_artifact_message_content` 类型映射的那次修正

我（Operator）最初的 prompt：

> `_artifact_message_content()` 要根据 `artifact.kind` 返回对应的 `content.type`，当前所有 artifact 都返回 `type: "code"`，但 kind=preview 的 artifact 在消息流里应该是 `type: "preview"`。

AI 第一版把所有 kind 都列了分支，但 `kind="diff"` 时返回的 schema 漏了 `fileName` 字段。Test 失败后我补充需求："Diff 卡片需要 fileName 让前端 DiffCard 渲染文件名"。

AI 补完后，我要求把这个映射逻辑写成注释，因为后续加新 `kind` 时容易漏分支。

> 反思：像 `_artifact_message_content` 这种"枚举→结构"的映射函数，最容易被以后的人改坏。应该写一个单测穷举所有 kind 值。这个单测目前缺失——列为 W5 待办。

### 3.2 Diff 应用的"双击冲突"问题

W4-Diff 落地当天，我测试连续点两次"应用"按钮：第一次成功创建了新版本，第二次因为 base 已经有子版本，返回了 `409 artifact_conflict`。但前端的错误提示只有"版本已过期，请刷新"，没有告诉用户第一次已经成功了。

我跟 AI 讨论了两条路：

- **A 方案**：前端点完按钮后立刻 disable，等 `artifact_ready` 回来再 enable。
- **B 方案**：后端做幂等，同一条消息的同一个 diff 不重复应用。

选 A，因为 B 方案会让后端"猜"前端的意图（同一条消息可能用户确实想 apply 两次不同修改）。A 方案 3 行代码搞定：`const [applying, setApplying] = useState(false)`。
