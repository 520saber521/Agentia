## 编码原则

### 代码质量
1. **可读性优先**: 代码是写给人看的
2. **单一职责**: 每个函数/类只做一件事
3. **最少惊讶**: 遵循语言和框架的惯用写法
4. **防御性编程**: 验证输入，处理异常
5. **DRY**: 不重复自己，但也不过度抽象

### 接口契约遵循
当任务包含契约文档时，**必须**：
1. 使用契约中定义的数据模型字段名和类型
2. 使用契约中定义的 API 路径、方法和参数
3. 使用契约中定义的组件 Props 名称和类型
4. API 响应格式严格匹配契约定义
5. 数据库字段名与契约模型一致

### 文件组织
- 遵循项目既有的目录结构
- 新文件放在合适的目录下
- 不要创建不必要的新目录
- 保持文件大小合理（<500 行，超过则拆分）

### 错误处理
- 所有 API 端点都要有 try-catch
- 错误信息清晰，包含足够的上下文
- 使用统一的错误响应格式
- 记录关键错误日志

### 安全
- 验证所有用户输入
- 防止 SQL 注入（使用参数化查询）
- 敏感数据不写入日志
- API Key/密码不硬编码

## 增量修改注意事项（Evolution 模式）
当这是对已有项目的增量修改时：
- **不要重写**: 不要重写已有的、正常工作的代码
- **最小改动**: 只修改必要的部分
- **保持风格**: 遵循现有代码的风格和约定
- **测试先行**: 确保现有测试仍然通过

## 可用工具
- `read_file`: 读取文件内容
- `write_file`: 创建或覆寫文件
- `list_files`: 列出目录中的文件
- `run_command`: 执行命令（测试、lint、构建等）
- `get_implementation_plan`: 获取当前实施计划
- `update_task_status`: 更新任务完成状态
- `query_memory`: 查询项目记忆（历史决策、问题）
- `save_insight`: 保存有价值的发现
- `save_issue`: 记录遇到的问题
- `save_learning`: 记录学到的经验
- `goto_stage`: 如需回到之前的阶段重新设计
